## 1. 제품 개요

### 1.1 제품명
**PatentAI** — AI 기반 특허 명세서 자동 생성 플랫폼

### 1.2 핵심 가치 제안
아이디어 문서(PRD), 논문(PDF), 자유 기술(텍스트)을 입력받아 **한국 특허청(KIPO) 표준 형식**의 특허 명세서를 단계별로 자동 생성하는 플랫폼. 연구자·산학협력단·변리사가 초안 작성 시간을 대폭 단축하고, AI가 분석한 특허 문장 패턴을 적용해 완성도 높은 명세서를 만들 수 있도록 지원한다.

### 1.3 MVP 목표
- **검증 목표:** AI 초안이 변리사의 실제 작업 시간을 50% 이상 단축하는지 확인
- **성공 지표:**
  - 입력 → 명세서 초안 완성까지 **30분 이내**
  - 생성된 청구항의 변리사 수용률 **60% 이상** (수정 없이 사용)
  - 초기 파일럿 10팀(연구실/산학협력단) 만족도 **4.0/5.0 이상**

### 1.4 대상 사용자

| 역할 | 설명 | 핵심 니즈 |
|------|------|-----------|
| **연구자** | 대학/연구소 교수·연구원 | 논문·연구결과를 특허로 전환, 명세서 초안 작성 부담 경감 |
| **산학협력단 담당자** | 대학 산학협력단 기술이전 담당 | 다수 연구 결과 특허화, 형식 검토 효율화 |
| **예비 변리사 / 변리사** | 특허 명세서 전문 작성자 | AI 초안 기반 편집·보정, 반복 작업 자동화 |
| **스타트업 / 발명자** | 기술 아이디어를 보유한 창업자 | 비용·시간 절감, 특허 지식 없이도 초안 생성 |

### 1.5 입력 유형 (3가지)

| 유형 | 입력 형식 | 예시 |
|------|-----------|------|
| **아이디어 기반** | 자유 텍스트 + PRD.md 파일 업로드 | CASA PRD.md → 창업 멘토링 AI 특허 |
| **논문 기반** | PDF 파일 업로드 | COOCON_Paper.pdf → LLM 스크래핑 특허 |
| **혼합 기반** | 텍스트 + PDF 복합 업로드 | PRD + 관련 논문 동시 분석 |

## 2. 특허 명세서 구조 (KIPO 표준)

### 2.1 한국 특허 명세서 표준 섹션

```
┌─────────────────────────────────────────────────────────────────┐
│               KIPO 표준 특허 명세서 구조                         │
├─────────────────────────────────────────────────────────────────┤
│  S1. 발명의 명칭          → 한국어 + 영문 제목                   │
│  S2. 기술분야             → 발명이 속하는 기술 도메인            │
│  S3. 배경기술             → 기존 기술의 문제점·한계              │
│  S4. 발명이 해결하려는 과제 → 본 발명의 목적 및 해결 과제       │
│  S5. 과제의 해결 수단     → 청구항 요약 형태의 해결 수단        │
│  S6. 발명의 효과          → 기술적·경제적 효과 기술             │
│  S7. 도면의 간단한 설명   → 도면 번호 및 간단 설명              │
│  S8. 발명 실시를 위한 구체적 내용 → 참조번호 기반 상세 설명    │
│  S9. 청구범위 (Claims)    → 독립항 1개 이상 + 종속항           │
│  S10. 요약서              → 200자 이내 요약                     │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 청구범위 작성 패턴 (분석 기반)

명세서 샘플 분석에서 추출한 핵심 문장 패턴:

#### 독립 청구항 (Independent Claim) 패턴
```
청구항 1
[발명 제목]로서,
[주요 구성요소 1]을 포함하는 [구성요소 1 설명];
[주요 구성요소 2]을 포함하는 [구성요소 2 설명];
...
을 포함하는 것을 특징으로 하는 [장치명/방법명].
```

#### 종속 청구항 (Dependent Claim) 패턴
```
청구항 N
청구항 [M]에 있어서,
상기 [구성요소]는
[추가 기능/세부 사항]을 더 포함하는 것을 특징으로 하는 [장치명/방법명].
```

#### 참조번호 체계 (Reference Numeral Convention)
```
100번대: 메인 장치/시스템 (100: 메인 장치)
  110-190: 1차 서브 모듈 (110: 입력부, 120: 처리부, ...)
    111-119: 2차 세부 컴포넌트 (111: 아이디어 입력 인터페이스, ...)
200번대: 서버/백엔드 시스템
  210-290: 서버 서브 시스템 (210: 스토리지, 220: 인증, ...)
```

#### 상세설명 주요 표현 패턴
| 용도 | 패턴 |
|------|------|
| 섹션 오프닝 | "본 발명은 ~에 관한 것이다." |
| 구성 요소 도입 | "상기 ~(110)은 ~역할을 한다." |
| 기능 기술 | "~를 포함하며, ~를 담당한다." |
| 구체화 | "구체적으로, ~는 ~를 수행한다." |
| 효과 서술 | "따라서, ~를 향상시킬 수 있다." |
| 도면 참조 | "도 [N]에 도시된 바와 같이," |
| 예시 | "예를 들어, ~의 경우에는 ~한다." |

## 4. 기능 요구사항 상세

### STEP 1: 입력 및 AI 분석 (F1)

#### F1-1. 입력 유형 선택
- 아이디어 텍스트 직접 입력 (최대 5,000자)
- PRD.md / 기획문서 파일 업로드 (.md, .txt, .docx)
- 논문 PDF 업로드 (.pdf, 최대 50MB, OCR 지원)
- 복합 입력: 텍스트 + 파일 혼합

#### F1-2. AI 기술 분석 (SSE 스트리밍)
AI가 입력을 분석하여 다음을 추출:
- **기술 도메인**: 발명이 속하는 기술 분야 분류 (IPC 코드 추천)
- **핵심 발명 포인트**: 기존 기술 대비 새로운 점 3~5개
- **해결 과제**: 기존 기술의 문제점 및 본 발명의 목적
- **기술적 효과**: 발명으로 인해 얻어지는 효과
- **구성요소 후보**: 참조번호 체계로 구조화된 컴포넌트 목록

#### F1-3. IPC 분류 자동 추천
- 입력 내용 기반 IPC(국제특허분류) 코드 상위 3개 추천
- 사용자가 선택 또는 직접 입력 가능

### STEP 2: 기술 구조화 (F2)

#### F2-1. 구성요소 자동 추출 및 참조번호 부여
- 발명의 주요 구성 모듈 자동 식별
- KIPO 표준 참조번호 체계(100번대/200번대)로 자동 매핑
- 계층적 컴포넌트 트리 생성 (메인 장치 → 서브 모듈 → 세부 기능)

#### F2-2. 구성요소 시각화
- 트리 구조로 컴포넌트 계층 표시
- 각 컴포넌트별 기능 설명 AI 생성
- 드래그앤드롭으로 사용자 재배치 가능

#### F2-3. 도면 구성 계획
- 필요한 도면 수 및 종류 자동 제안
  - 시스템 구성도 (System Architecture)
  - 흐름도 (Flowchart)
  - UI 레이아웃 (선택)
  - 데이터 흐름도 (선택)

#### F2-4. 선행기술조사 (GATE 2 연동)

구성요소가 확정되는 시점인 GATE 2에서 선행기술조사를 함께 수행한다. 청구항 작성 전에 유사 특허를 파악해야 **구성요소 레벨에서 차별화 포인트를 조정**할 수 있기 때문이다.

**검색 대상 데이터베이스:**
| DB | 대상 | API |
|----|------|-----|
| **KIPRIS** | 국내 특허 (주) | KIPRIS Open API (무료) |
| **USPTO** | 미국 특허 | USPTO PatentsView API (무료) |
| **Google Patents** | 글로벌 | 향후 확장 |

**AI 유사도 분석 방식:**
1. 구성요소 키워드 + IPC 코드 기반 KIPRIS 자동 검색
2. 상위 10건 반환 → 청구항 임베딩 벡터 유사도 계산
3. 구성요소별 충돌 위험도 산출 (높음/중간/낮음)
4. 충돌 위험 구성요소에 AI 회피 전략 제안

**UI 동작 규칙:**
- 유사 특허 없음 → 패널 자동 접힘, "✅ 유사 특허 없음" 표시
- 유사 특허 발견 → 패널 자동 펼침, 위험 구성요소 하이라이트

| **선행기술 패널** | 유사 특허 없을 시 자동 접힘 / 유사 특허 발견 시 자동 펼침 |

---

### STEP 3: 청구범위 초안 작성 (F3)

**청구범위(Claims)는 특허의 핵심이므로 별도 단계로 분리한다.**

#### F3-1. 독립항 생성 (Claim 1)
- 구성요소 구조를 기반으로 "~를 포함하는 것을 특징으로 하는 장치/방법" 형식 생성
- 장치 청구항 / 방법 청구항 / 시스템 청구항 중 유형 선택 가능
- KIPO 표준 문장 패턴 적용
- **GATE 2 선행기술 조사 결과를 컨텍스트로 활용**: 충돌 위험 구성요소는 차별화 표현으로 자동 강조

#### F3-2. 종속항 생성 (Claim 2~N)
- 각 서브 모듈별 종속항 자동 생성
- "청구항 N에 있어서, ~를 더 포함하는" 패턴 적용
- 권장 종속항 수: 5~15개 (독립항 복잡도에 따라 조정)

#### F3-3. 청구항 강도 분석
- AI가 각 청구항의 **특허 등록 가능성** 평가
  - 신규성(Novelty) 위험도
  - 진보성(Inventive Step) 강도
  - 명확성(Clarity) 점수
- 개선 제안 함께 제공

#### F3-4. 청구항 편집기
- 각 청구항을 인라인 편집 가능
- 청구항 추가 / 삭제 / 순서 변경
- 종속 관계 시각화 (청구항 트리)

### STEP 4: 명세서 본문 작성 (F4)

청구범위 확정 후 나머지 8개 섹션을 AI가 순차 생성한다.

#### F4-1. 섹션별 생성 순서 (SSE 스트리밍)
```
1. 발명의 명칭       → 한·영 제목 생성
2. 기술분야          → IPC 기반 기술 도메인 기술
3. 배경기술          → 기존 기술 문제점 3~5개 단락
4. 해결하려는 과제   → 발명 목적 서술
5. 과제의 해결 수단  → 청구항 요약 재기술
6. 발명의 효과       → 기술적·경제적 효과 3~5개
7. 도면의 간단한 설명 → 도면 번호별 한 줄 설명
8. 발명 실시를 위한 구체적 내용 → 참조번호 기반 상세 설명
9. 요약서            → 200자 이내 요약
```

#### F4-2. 상세설명 작성 품질 기준
- 각 구성요소(참조번호)는 **최소 2개 이상의 단락**으로 기술
- "상기 ~(110)은 ~역할을 한다." 표준 패턴 준수
- 도면 참조 표현 자동 삽입: "도 N에 도시된 바와 같이,"
- 기술 용어 일관성 자동 검사

#### F4-3. 섹션 개별 재생성
- 섹션별 독립적 재생성 가능
- 특정 섹션 지정 수정 요청 (인라인 채팅으로 지시)

## 7. AI 프롬프트 엔진 설계

### 7.1 분석 프롬프트 (STEP 1)
```
역할: 당신은 한국 특허 전문가이자 기술 분석가입니다.
임무: 아래 [입력 문서]를 분석하여 특허 명세서 작성에 필요한 핵심 기술 정보를 추출하세요.

출력 형식 (JSON):
{
  "tech_domain": "기술 분야 설명",
  "ipc_suggestions": ["G06F 40/56", "G06N 20/00"],
  "core_inventions": ["핵심 발명 포인트 1", ...],
  "problems_solved": ["해결 과제 1", ...],
  "effects": ["기술적 효과 1", ...],
  "key_components": [{"name": "구성요소명", "function": "기능 설명"}, ...]
}
```

### 7.2 선행기술 회피 전략 프롬프트 (GATE 2)
```
역할: 당신은 한국 특허 전문가입니다.
임무: 아래 [유사 특허 목록]과 [구성요소 트리]를 비교하여 충돌 위험을 분석하고 회피 전략을 제안하세요.

분석 항목:
1. 각 유사 특허의 청구항과 현재 구성요소 간 중복 여부
2. 충돌 위험이 높은 구성요소 식별 (risk_level: high/medium/low)
3. 각 충돌 구성요소에 대한 구체적 차별화 방안 제시

출력 형식 (JSON):
{
  "conflicting_components": [
    {
      "ref_number": "120",
      "component_name": "AI분석부",
      "conflicting_patent": "KR10-2024-0012345",
      "risk_level": "high",
      "avoidance_suggestion": "멀티모달 입력 처리 기능을 추가하여 차별화"
    }
  ],
  "overall_risk": "medium",
  "summary": "전반적 위험도 요약 및 권장 조치"
}
```

### 7.3 청구항 생성 프롬프트 (STEP 3)
```
역할: 당신은 한국 특허청(KIPO)에 출원 경험이 풍부한 변리사입니다.
임무: 아래 구성요소 구조를 기반으로 KIPO 표준 청구범위를 작성하세요.

규칙:
- 독립항은 반드시 1개 이상 작성
- 종속항은 "청구항 N에 있어서" 로 시작
- 모든 청구항은 "~을 특징으로 하는 [발명 유형]."으로 종료
- 모호한 표현(최선, 최고, 약, 대략) 사용 금지
- 참조번호를 (110) 형식으로 반드시 포함
```

### 7.4 도면 생성 프롬프트 (STEP 5 - Gemini)
```
Patent technical drawing, black and white line art, clean white background,
no color fills, professional engineering diagram style,
[도면 유형: system architecture / flowchart / UI wireframe],
labeled with reference numerals in format (100), (110), (120),
boxes connected by arrows, KIPO patent filing quality,
300 DPI, precise and minimal, no shadows or gradients
```

## 9. 사용자 인터페이스 요구사항

### 9.1 프로젝트 대시보드
- 진행 중인 특허 프로젝트 목록 (카드 형태)
- 각 프로젝트의 현재 단계 시각화 (STEP 1~6 진행바)
- 완성된 명세서 다운로드 버튼
- 새 프로젝트 시작 버튼

### 9.2 명세서 작성 뷰 (단계별 화면)
- 왼쪽: 현재 단계 작업 영역
- 오른쪽: 명세서 미리보기 (실시간 업데이트)
- 상단: 진행 단계 표시 (Step 1~6 + Gate 상태)
- AI 생성 중 SSE 스트리밍 텍스트 표시

### 9.3 청구항 편집기
- 청구항 목록 (독립항·종속항 구분 표시)
- 인라인 텍스트 편집
- 청구항 트리 시각화 (종속 관계 표시)
- 드래그앤드롭 청구항 순서 변경

### 9.4 도면 뷰어
- 도면 목록 (썸네일 그리드)
- 도면 클릭 시 전체화면 미리보기
- 프롬프트 확인 및 수정 후 재생성 버튼
- 외부 이미지 업로드 교체 기능

## 10. MVP 구현 우선순위

### Phase 1 (핵심 MVP — 4주)
- [ ] 프로젝트 생성 및 입력 처리 (F1)
- [ ] 기술 분석 AI 및 GATE 1 (F1-2)
- [ ] 청구범위 생성 및 편집 (F3)
- [ ] 명세서 본문 생성 (F4) — 기술분야·배경기술·청구항·요약 4개 섹션
- [ ] DOCX 출력 (F6)

### Phase 2 (완성도 향상 — 3주)
- [ ] 구성요소 트리 시각화 및 편집 (F2)
- [ ] **선행기술 조사 — KIPRIS API 연동 + 유사도 분석 (F2-4)**
- [ ] **선행기술 회피 전략 AI 제안 (GATE 2 통합)**
- [ ] 명세서 전체 9개 섹션 생성 (F4 완전)
- [ ] 도면 생성 (F5 — Gemini 연동)
- [ ] PDF 출력 (F6)
- [ ] 청구항 강도 분석 (F3-3)

### Phase 3 (품질·확장 — 3주)
- [ ] 논문 PDF OCR 입력 (F1)
- [ ] 명세서 완성도 리포트 (F6-3)
- [ ] 변리사 협업 기능 (공유·코멘트)
- [ ] 다국어 지원 (영어 명세서 생성)
- [ ] 선행기술 조사 확장 — USPTO / Google Patents 연동
