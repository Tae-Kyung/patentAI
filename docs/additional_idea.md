## 1. 신규 기능: 외부 도면 이미지 추가 및 해석

### 고객 의견 요약
- 업무상 고객이 외부 상세 도면(CAD, 기술 도면 등)을 추가하는 경우가 많음
- 현재는 AI가 도면을 새로 생성하는 것만 가능
- 기존 외부 도면을 업로드하고, 해당 도면을 설명하거나 간소화하여 재생성하는 기능 필요

### 분석
현재 STEP5(도면 생성)는 AI(Gemini)가 도면을 새로 생성하는 방식만 지원한다.
실제 업무에서는 이미 완성된 기술 도면(제품 설계도, CAD 도면, 논문 그림 등)을 활용하는 경우가 더 많기 때문에,
외부 도면을 특허 명세서에 통합하는 기능은 실용적 가치가 높다.

### 개선 방안

#### 1-A. 외부 도면 업로드 기능 (STEP5 확장)
- 도면 탭에 "직접 업로드" 옵션 추가 (PNG/JPG/SVG/PDF)
- 업로드된 도면을 `patentai_patent_drawings` 테이블에 `source: 'external'` 플래그로 저장
- Supabase Storage에 원본 이미지 보관

#### 1-B. 외부 도면 AI 해석 기능
- 업로드된 도면 이미지를 GPT-4o Vision 또는 Gemini Vision으로 분석
- 도면에서 구성요소, 참조번호 후보, 동작 흐름 자동 추출
- 추출된 설명을 S7(도면의 간단한 설명), S8(발명 실시를 위한 구체적 내용)에 자동 반영 제안

#### 1-C. 외부 도면 간소화 재생성 기능
- 업로드된 상세 도면을 특허 도면 스타일(선화, 단순 흑백)로 변환
- Gemini API에 원본 이미지를 참조로 제공하여 특허 규격 도면 재생성
- 원본 도면과 재생성 도면을 나란히 비교 표시하여 사용자가 선택

#### 구현 위치
- `features/patent/step5/` — 도면 업로드 UI 컴포넌트 추가
- `app/api/patents/[id]/drawings/external/` — 외부 도면 처리 API 라우트 신규
- `lib/ai/` — Vision 분석 래퍼 함수 추가

## 2. 버그 수정: 도면 이미지 하단 잘못된 Fig. 번호 추가

### 고객 의견 요약
- 도면 이미지 생성 시 하단에 잘못된 Fig. 번호 내용이 추가됨

### 분석
Gemini API에 전달되는 도면 생성 프롬프트에 Fig. 번호를 이미지 내부에 텍스트로 포함하도록 지시하는 내용이 있을 가능성이 높다.
특허 도면에서 Fig. 번호는 명세서 본문(S7)에서 참조하는 형태로 기술되어야 하며, 이미지 내부에 삽입되면 안 된다.

### 개선 방안
- Gemini 도면 생성 프롬프트에서 "Fig. X" 텍스트를 이미지 내부에 렌더링하지 말도록 명시적으로 지시 추가
- 프롬프트 수정: `"이미지 내부에 Fig. 번호, 제목, 텍스트 라벨을 포함하지 마시오. 도면 설명은 텍스트로 별도 제공한다."`
- `patentai_prompts` 테이블의 도면 생성 프롬프트 항목 수정

#### 구현 위치
- `app/api/patents/[id]/drawings/generate/route.ts` — 프롬프트 파라미터 검토
- 관리자 페이지 프롬프트 관리에서 도면 생성 프롬프트 직접 수정

## 3. 버그 수정: "KIPO 도상" 문구/라벨 추가 오류

### 고객 의견 요약
- 도면 이미지에 "KIPO 도상"이라는 문구나 라벨이 가끔 추가되는 오류 발생

### 분석
Gemini 모델이 학습 데이터에서 KIPO 특허 도면에 자주 등장하는 워터마크나 라벨 패턴을 모방하여 생성하는 것으로 추정된다.
이는 프롬프트 강화와 후처리 검증으로 해결 가능하다.

### 개선 방안
- 도면 생성 프롬프트에 명시적 금지 지시 추가: `"워터마크, 기관명, 'KIPO', '특허청', 도장, 로고 등 일체의 기관 식별 요소를 포함하지 마시오."`
- 생성 후 이미지 텍스트 검출(OCR) 또는 키워드 필터를 통한 후처리 검증 로직 추가 (선택적)
- 오류 발생 시 자동 재생성 트리거 (최대 2회 재시도)

#### 구현 위치
- 도면 생성 프롬프트 수정 (위 2번과 동일)
- `lib/ai/gemini.ts` — 이미지 생성 후 검증 로직 추가 검토

## 4. 버그 수정: 계획된 도면 수와 실제 생성 수 불일치

### 고객 의견 요약
- 최초 3개의 도면을 계획했는데 더 많은 도면이 그려짐
- 계획과 도면 생성 부분이 연결되어 있는지 확인 필요

### 분석
STEP2(기술 구조화)에서 도면 계획(몇 개, 어떤 도면)을 수립하지만, STEP5(도면 생성)에서 이 계획을 참조하지 않고
AI가 독립적으로 도면 수를 결정할 가능성이 있다. 계획 데이터와 생성 로직 간 연결이 끊어진 상태로 보인다.

### 개선 방안
- STEP2에서 확정된 도면 목록(`patentai_patent_drawings` 또는 관련 컬럼)을 STEP5 생성 API에서 반드시 참조하도록 강제
- 도면 생성 API 호출 시 `planned_count` 파라미터를 명시적으로 전달하고, Gemini 프롬프트에 `"정확히 N개의 도면만 생성하시오"` 지시 포함
- UI에서 계획된 도면 목록을 STEP5 진입 전에 사용자에게 확인시키는 검토 단계(GATE4 또는 STEP5 초입) 추가

#### 구현 위치
- `app/api/patents/[id]/drawings/generate/route.ts` — 계획 데이터 참조 로직 추가
- `features/patent/step5/` — 계획된 도면 목록 표시 UI

## 5. 버그 수정: 도면 재생성 시 진행 상태 미표시

### 고객 의견 요약
- 도면 재생성 버튼을 누르면 재생성 과정이 표시되지 않음 (사용자가 진행 중인지 알 수 없음)

### 분석
도면 최초 생성 시에는 SSE 스트리밍으로 진행 상태를 표시하더라도, 재생성(regenerate) 경로에서는 로딩 상태 관리가 누락된 것으로 추정된다.

### 개선 방안
- 재생성 버튼 클릭 시 해당 도면 카드에 로딩 스피너 또는 진행 인디케이터 표시
- SSE 스트리밍 패턴 적용: 재생성 API도 `createSSEResponse`를 통해 진행 이벤트(`generating`, `done`, `error`) 발행
- 재생성 중에는 버튼 비활성화 처리하여 중복 클릭 방지

#### 구현 위치
- `features/patent/step5/` — 재생성 버튼 상태 관리 및 로딩 UI 추가
- `app/api/patents/[id]/drawings/[drawingId]/regenerate/route.ts` — SSE 응답 방식으로 전환

## 6. 버그 수정: 프로젝트 목록 중복 카드 현상

### 고객 의견 요약
- 새로운 프로젝트 추가 시 중복된 카드가 나타나는 현상 발생

### 분석
프로젝트 생성 API 호출 성공 후 UI 상태 업데이트 방식에 문제가 있을 가능성이 높다.
- Optimistic update(낙관적 업데이트)와 실제 API 응답 후 refetch가 동시에 일어나거나
- 이벤트 중복 발생(버튼 더블클릭, 빠른 재요청)으로 DB에 중복 레코드가 삽입되는 경우

### 개선 방안

#### 6-A. 중복 생성 방지 (API 레벨)
- 프로젝트 생성 API에 멱등성 보장: 동일 사용자+동일 제목+짧은 시간 내 중복 요청 차단
- 생성 버튼 클릭 후 응답 수신까지 버튼 비활성화 처리

#### 6-B. 상태 관리 정합성 (UI 레벨)
- 프로젝트 목록 조회를 API 응답 완료 후 단 한 번만 실행하도록 수정
- Optimistic update 사용 중이라면 API 응답 후 서버 데이터로 덮어쓰기(replace) 처리

#### 구현 위치
- `app/api/patents/` 또는 `app/api/projects/` — 중복 생성 방지 로직
- `features/patent/` 또는 `app/[locale]/(dashboard)/dashboard/` — 프로젝트 목록 상태 관리
